Say it as: “Chas-Plums”. Imagine Claude eating plums while navigating a huge codebase. Each letter is one part of the Claude Code harness.

C

CLAUDE.md

Context files loaded automatically. Root gives the big picture; subdirectories give local conventions.

Memory image: a map taped to every doorway.
H

Memory image: hooks catching mistakes before they fall.
S

Skills

On-demand expertise. Load specialized workflows only when the task needs them.

P

Plugins

Bundles of skills, hooks, and MCP configs. They distribute working setups across the org.

Memory image: a plug powering every engineer’s desk.

Memory image: a bridge from Claude to company systems.
S

Subagents

Separate Claude instances that explore, summarize, or work in parallel without polluting the main context.

Memory image: scout agents returning with maps.

2. Second Acronym: “LIVE Beats STALE”

Use this to remember why the blog prefers Claude Code’s agentic navigation over a pure RAG index for large, fast-changing repos.

LIVE

Agentic Search

Local codebase, Immediate filesystem state, Version-current files, Engineer-like navigation.

Remember: Claude walks the current repo like a developer.
STALE

Index Risk

Snapshotted embeddings, Time lag, Archived references, Lost renames, Expired modules.

Remember: stale indexes may return yesterday’s code.

3. Memory Palace: Walk Through the Enterprise Codebase Building

Imagine entering a giant engineering office. Each room represents one deployment best practice. Walk the rooms in order to recall the whole article.

🚪

Entrance: Start with CLAUDE.md

Root = big picture. Subdirectory = local conventions.

Memory cue: Before Claude can work, it needs a building directory. Keep the entrance sign lean: pointers, gotchas, and conventions only.

🧭

Hallway: Scope Claude to the Right Area

Start in relevant subdirectories, not always repo root.

Memory cue: Don’t tour the entire city to fix one room. Claude walks upward and still sees parent context when needed.

🧪

Test Lab: Localize Test and Lint Commands

Run only relevant checks for the changed service.

Memory cue: A small experiment belongs in the right lab. Full-suite output wastes time and context.

🗑️

Storage Room: Ignore Noise

Exclude generated files, build artifacts, and third-party code.

Memory cue: Claude should not search trash piles unless the task is about trash piles. Use project-level exclusions with local overrides when needed.

🗺️

Map Room: Build Codebase Maps

Useful when directory structure is not self-explanatory.

Memory cue: A root-level markdown table of contents helps Claude know where to look first.

🎯

Control Room: Add LSP

Search by symbol, not just by string.

Memory cue: LSP is the sniper scope. It helps Claude find the exact function, class, or reference.

🔁

Maintenance Room: Review Config Every 3–6 Months

Old instructions can become constraints as models improve.

Memory cue: Clean the house after every major model upgrade. Remove instructions that only existed to compensate for older limitations.

👥

Org Room: Assign Ownership

Developer productivity, agent manager, or DRI.

Memory cue: Without an owner, good setups become tribal knowledge. With an owner, patterns become organization-wide defaults.

6. Self-Quiz: Active Recall

Try answering each question before opening it. This is the fastest way to move from recognition to memory.

Q1. What is the article’s main thesis?

The model is not enough. Claude Code performs well in large codebases when the surrounding harness—CLAUDE.md, hooks, skills, plugins, LSP, MCP, and subagents—is designed well.

Q2. Why can a RAG-style codebase index fail in large active repos?

The index can become stale because large teams change code constantly. Retrieval may return renamed, deleted, or outdated functions.

Q3. What should go into root CLAUDE.md?

Only broad, durable information: big-picture structure, critical gotchas, conventions, and pointers. Avoid overloading it with narrow workflows.

Q4. Why start Claude in a subdirectory?

It scopes the task to the relevant part of the codebase while still allowing Claude to walk upward and load parent CLAUDE.md context.

Q5. What is the difference between a skill and a CLAUDE.md file?

CLAUDE.md is always loaded as baseline context. A skill is specialized expertise loaded on demand only when relevant.

Q6. Why is LSP high value for large codebases?

It gives symbol-level navigation, helping Claude find the correct definition or reference instead of opening many irrelevant grep matches.

Q7. What organizational role is needed for successful adoption?

A DRI, developer productivity team, or agent manager who owns conventions, permissions, plugins, governance, and ongoing maintenance.
